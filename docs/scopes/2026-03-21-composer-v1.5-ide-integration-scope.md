# Scope: Composer v1.5 — IDE Integration

## Problem
The platform has strong research capabilities (10-tool assistant, 7 analyses, ML prediction) and a functional composer, but they exist on separate pages. A policy analyst must leave the drafting workspace to research, then manually bring findings back. This gap is what separates "AI text editor" from "integrated drafting environment." Closing it is the highest-leverage move toward the "Cursor for public policy" vision.

## In Scope
- Workspace-aware assistant panel embedded in the composer detail page
  - Chat backend variant that injects workspace context (goal, jurisdiction, precedents, section headings, current drafts) into the system prompt
  - All 10 existing research tools available within the workspace conversation
  - Conversation scoped to workspace (persisted, resumable)
- Draft-aware analysis actions in the composer
  - "Analyze my draft" actions that run constitutional and pattern-detection prompts against user-drafted text (not bill_id lookups)
  - Results displayed inline with section content
- Fix P2 #136 (DB connection held during LLM calls) — will crash multi-compose demos
- **Selective expansion 1:** Precedent insight cards — show ML prediction probability + AI summary for each precedent bill inline in the workspace (data already exists, just surface it)
- **Selective expansion 2:** "Apply suggestion" flow — when the assistant recommends specific language or a precedent, one-click to feed it into a compose action for the active section

## Out of Scope
- Real authentication / user accounts — X-Client-Id sufficient for demo stage
- Rich editor framework (TipTap/Lexical/ProseMirror) — textarea + compose/accept proves the concept
- Multi-model routing / OAuth for frontier models — Claude-only is fine for now
- Collaboration / multi-user drafting — single-user demo
- Plugin/extension architecture — premature
- Remaining P1 security backlog (prompt injection, CSV sanitization, model artifact integrity) — production concerns, not demo blockers
- Word/PDF export — markdown is sufficient for stakeholder demo
- State-level ML prediction expansion — federal-only model is impressive enough

## Key Constraints
- Chat backend already exists (753-line chat.py) — extend, don't rebuild
- Research tools already defined (tools.py, 10 tools) — reuse as-is, add workspace-aware variants
- Composer data model is stable (5 tables, just shipped) — no schema changes to workspace/section/generation tables
- Frontend composer detail page is already ~920 lines — consider extracting the assistant panel as a separate component rather than growing the monolith
- LLM harness methods that accept bill_id need parallel variants that accept raw text for draft analysis

## Codebase Context
- src/api/chat.py — existing chat with tool dispatch and agentic loop; needs workspace-context variant
- src/llm/tools.py — 10 research tools; workspace-scoped tools can wrap these with context
- src/llm/harness.py — analysis methods take bill_id; new methods needed for raw text input
- src/services/policy_composer_service.py — composer business logic; draft analysis fits here
- frontend/src/app/composer/[id]/page.tsx — already has a minimal "research panel" prototype to build on
- src/prediction/service.py — predict_bill() for precedent insight cards

## Open Questions
- Should workspace assistant conversations persist across sessions, or reset each visit?
- Should draft analysis results be stored in policy_generations (for audit trail) or ephemeral?
- What's the right UX for "apply suggestion" — auto-populate compose form, or create a pending generation directly?
